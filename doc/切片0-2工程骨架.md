## 切片 0 + 2 工程骨架（可直接按模块开工）

> 目标：尽快做出“可验证的策略体验”与“多人同花园同步”的最小闭环。
>  
> - 切片 0：连续游戏时钟 + 离线不暂停 + 离线摘要（回归后指导下一步）
> - 切片 2：同花园多人存在 + WebSocket 实时同步 + 至少 2~3 个动作（`Feed/Cuddle/Pat`）的状态策略反馈

---

## 1. 协议先行（必须做，不然前后端对不上）
后端对所有状态更新拥有权威；前端只负责发送动作与渲染展示。

### 消息类型（按你的设计文档）
1. `ClientToServer`
   - `joinGarden`：进入花园，拉取快照
   - `updatePointer`：鼠标投影坐标（节流）
   - `petAction`：喂食/抱抱/摸头（携带动作参数）
2. `ServerToClient`
   - `gardenSnapshot`：进入时的初始状态
   - `pointerUpdate`：他人指针坐标
   - `actionBroadcast`：他人动作导致的动画事件（用于他人可见反馈）
   - `petStateDelta`：宠物状态增量（用于本地状态条更新）
   - `eventBroadcast`：事件开始/结束与任务提示（切片 2 可先只占位）

### 协议实现要点
1. 所有消息都带 `requestId`（可选但强烈建议，用于前端匹配响应/排错）
2. `petAction` 必须携带：
   - `actionType`
   - `petId`
   - `itemId?`（喂食需要）
3. 后端必须做校验：
   - 用户是否属于该 `petId`
   - 花园房间是否匹配
   - 物品是否存在且数量充足（切片 2 可先简化：允许“演示道具”或先跳过扣库存）

## 2. 切片 0：连续游戏时钟 + 离线摘要

### 后端模块建议
1. `backend/game/time.py`
   - 管理服务器开服时间 `serverStartWallClock`
   - 提供函数：
     - `getGameTime(nowWallClock) -> {gameDayIndex, gameHourIndex, gameHourFloat}`
     - `realHoursToGameHours(realHours) = realHours * 2`（由设计：现实 1 小时 = 游戏 2 小时）
2. `backend/game/offline.py`
   - 负责离线差异计算与摘要生成规则（不需要 AI，先规则化）
   - 输入：宠物状态在“离开时刻游戏时间”与“当前游戏时间”的差异
   - 输出：1~2 条离线摘要原因 + 当前建议动作类型（例如 `Feed` 或 `TreatAtHospital`）

### 核心算法口径（写清楚即可实现）
1. 让游戏世界连续流动的实现方式：
   - **不要**做“房间在线才推进时间”
   - 每次需要结算（例如进入花园、每个 tick 或每次动作后回算），都用墙钟时间换算当前游戏时间
2. 离线摘要生成（MVP 可简化但要可解释）：
   - 比较离开时刻到当前时刻的关键指标变化：
     - `hunger` 是否显著上升（导致健康趋势下降）
     - `sick` 是否进入过“出现过”的状态
     - `mood` 是否持续下降
   - 将“变化最大的 1~2 项原因”落成摘要文案
   - 给出 1 条建议动作：优先 `Feed`，若健康很低则 `TreatAtHospital`

### 切片 0 UI 必要 DOM（HTML 叠层）
1. `OfflineSummary` 面板（登录/进入花园立即弹出）
   - 标题：`离线摘要`
   - 原因：1~2 条 bullets（来自离线摘要）
   - 建议动作按钮：`去喂食` 或 `去医院`（点后直接触发对应动作界面/按钮）

## 3. 切片 2：多人同花园 WebSocket 同步 + 状态策略可玩

### 后端模块建议
1. `backend/realtime/garden_ws.py`
   - WebSocket 连接管理
   - `Garden` 作为房间维度，负责广播：
     - `pointerUpdate`
     - `actionBroadcast`
     - `petStateDelta`
2. `backend/game/actions.py`
   - 实现动作处理入口：
     - `handleFeed(pet, foodItem)`
     - `handleCuddle(pet)`
     - `handlePat(pet)`
   - 每个动作返回：
     - 状态变更结果（hunger/health/mood/loyalty/sickLevel 等）
     - 需要广播的动画事件字段
3. `backend/game/tick.py`
   - 每个 `GameDay` 结算逻辑（hunger/health/mood/sick 判定）
   - 成长阶段推进逻辑（情绪稳定优先的 stabilityScore 与回退）

### 客户端流程（前端 HTML 叠层 + Phaser）
1. 进入花园：
   - WebSocket 连接成功
   - 发送 `joinGarden`
   - 接收 `gardenSnapshot`
   - 同步渲染：
     - Phaser：宠物小屋、宠物精灵、他人指针（初始）
     - HTML：状态条、风险提示、成长面板、动作按钮
2. 鼠标投影：
   - 本地鼠标移动时更新 Phaser 指针位置
   - 节流发送 `updatePointer`
3. 动作：
   - 点击 HTML 按钮触发本地 UI feedback（例如 loading/按钮禁用）
   - 发送 `petAction`
   - 等待服务器广播回来的 `petStateDelta` + `actionBroadcast`
   - 更新本地状态条并播放动画（他人动作也同理）

### 切片 2 UI 必要 DOM
复用你已生成的图对应结构：
1. 花园主界面（已生成）
   - 动作按钮：`喂食` / `抱抱` / `摸头`（切片 2 至少 3 个）
   - 金币 HUD（可以先占位）
   - 状态条：hunger/health/mood
   - 风险提示：可能生病/建议下一步
2. 成长面板（已生成）
   - `growthStage`
   - `stabilityScore`
   - “距离下一阶段还差什么”最多 1~2 条缺口

## 4. 最小验收清单（用来判断切片 0+2 是否“真的能玩”）
1. 切片 0（离线不暂停）：
   - 玩家离开后过一段时间，再进入花园：
     - 状态条已随时间变化（不因在线与否暂停）
     - 离线摘要出现且能指导下一步（Feed 或 TreatAtHospital）
2. 切片 2（多人同步可感知）：
   - 两个用户在同一花园：
     - 任一用户点击动作，另一用户在 0.5~1 秒内看到状态条/动画/指针变化至少一种
3. 策略体验成立：
   - 在“建议动作”下，状态变化应当比随机动作更有效（至少对 `sick` 风险形成可感知差异）
