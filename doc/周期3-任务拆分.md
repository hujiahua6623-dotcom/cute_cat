## 1. 已定稿口径（本次确认）

### 1.1 事件列表的 SSOT

- **是**：当前花园下**正在进行或对本客户端可见**的事件，以 **`gardenSnapshot.payload.activeEvents`** 为**第一眼真值**（SSOT）。客户端进房渲染 UI 时**必须**以快照中的数组为准初始化。
- **`eventBroadcast`**：用于 **增量更新**（进度变化、阶段切换 `started` → `tick` → `ended`、补发提示等）。客户端逻辑：**快照覆盖 + 按 `eventId` 合并广播**。

### 1.2 生日周年（一维 `gameDayIndex` 下可实现的规则）

设计语义仍为：**领宠当日写入的 `birthdayGameDay` 为「生日锚点」**；之后**每逢一个「游戏年」**在对应游戏日重演一次生日活动。

- **游戏年长**：常量 **`YEAR_GAME_DAYS = 365`**（与真实历法无关；后续若引入「游戏年月日」可再替换，届时同步改本文档与测试）。
- **判定当前游戏日是否为该宠物的周年当日**：  
  `game_day_index >= birthday_game_day`  
  且 `(game_day_index - birthday_game_day) % YEAR_GAME_DAYS == 0`
- **持续时间**：与产品一致——**恰好 1 个 `GameDay`（24 游戏小时）**。即：仅在「周年当日」的 `game_day_index` 一整天内，`BirthdayEvent` 处于激活状态（由当前 wall clock 推得的 `gameTime.gameDayIndex` 是否等于该日判定）。

> 说明：领宠时后端已将 `birthday_game_day` 设为当时 `game_day_index`（见 `POST /pets/claim`），故**首年**生日即为领宠日当日，无需冷启动特例。

### 1.3 花园社交事件——MVP 可复现调度（占位策略）

在未上「全服统一随机窗口」前，使用**确定性公式**，便于联调与单测：

- 常量：`SOCIAL_PERIOD_GAME_DAYS = 10`，`SOCIAL_SPAN_GAME_DAYS = 2`（连续 2 个 `GameDay`，合计 48 游戏小时，落在产品「12～48 游戏小时」区间内；模板化后可改为读配置）。
- `phase = int(sha256(garden_id.encode("utf-8")).hexdigest(), 16) % SOCIAL_PERIOD_GAME_DAYS`（实现时与后端 `hashlib.sha256` 一致）。
- 当 **`((game_day_index - phase) % SOCIAL_PERIOD_GAME_DAYS) < SOCIAL_SPAN_GAME_DAYS`** 时，认为该 `gardenId` 在当前 `game_day_index` **处于花园社交事件窗口内**。

不同花园 `phase` 不同，不同步开场；后续若要求「全服统一」再改为单一种子或运营配置。
