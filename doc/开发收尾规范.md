## 开发收尾规范（每日/每次会话结束）

用于规范每天开发结束后的统一动作，确保文档与代码状态持续同步，便于多人协作和后续开源维护。

### 1. 必做清单（每次结束前）
1. 更新开发进度日志：`doc/开发进度日志.md`
   - 新增当日小节（日期）
   - 填写：
     - 今日完成（按模块/功能）
     - 涉及文件（路径）
     - 已知问题/风险
     - 下一步计划（最多 3 条）
2. 技术栈变更同步
   - 若有技术选型、依赖、架构变化，必须同步：
     - `README.md`（技术栈概览）
     - `doc/项目目录说明.md`（结构与栈说明）
3. 规则与行为变更同步
   - 若改动游戏规则、协议、事件流程、成长逻辑：
     - 同步 `doc/项目设计文档.md`
     - 必要时同步 `doc/切片0-2工程骨架.md`
4. 资源索引同步
   - 新增/替换/删除 UI 或文档时，更新 `doc/项目目录说明.md` 的对应条目。

### 2. 自动更新要求（Agent 执行）
1. 任何实质改动结束后，Agent 应主动检查是否触发以下更新：
   - 是否需要补写当日进度日志
   - 是否需要同步技术栈说明
   - 是否需要同步设计与骨架文档
   - 是否需要同步目录资源索引
2. 若触发条件满足，Agent 应在同一会话内完成对应文档更新，不留“后补”。

### 3. 触发条件（何时必须更新）
1. 技术栈/框架变化（例如新增 LangChain、替换数据库）
2. 核心行为变化（例如时间换算、成长规则、事件周期）
3. 协议变化（WebSocket 消息结构字段有改动）
4. 资源变化（新增/重命名/替换 UI 图）

### 3.1 防复发检查（必须逐项打勾）
- [ ] `README.md` "Current Phase Card" 已与当前状态一致
- [ ] `AGENTS.md` 阶段描述与日志最新结论一致
- [ ] 最新日志含“可追溯证据”（构建/测试/脚本/人工走查至少一项）
- [ ] 所有已知风险都带“解决周期”或“触发条件”
- [ ] 日志写明“下一步唯一优先动作”（owner + scope）

### 4. 日志模板（复制追加）
```markdown
## YYYY-MM-DD

### 今日完成
1. ...

### 涉及文件
- `path/to/file`

### 已知问题/风险
- ...

### 下一步计划
1. ...

### 防复发检查（收尾）
- [ ] 口径一致（README/AGENTS/日志）
- [ ] 验收证据已记录
- [ ] 风险已落地到周期或触发条件
- [ ] 下一步唯一优先动作已明确
```
